Step 2 — Create 3 Specialised Agents

Design three agents to help with onboarding new projects. Each agent skill is stored under .bob/skills/.

2.1 Documentation Agent

Specialises in onboarding a new project — checks all files thoroughly and notes its findings.

2.2 Code Quality Critic

Reviews code quality across the project and surfaces issues.

2.3 Bug Finding Agent

Finds bugs in the project and proposes or applies fixes.

Section 1 — Token Consumption

  1. 1
    Say "hello" to Bob

    Open Bob and type hello. Can you spot something strange? Does the token consumption make sense to you?

  2. 2
    Think about optimisation

    Observe the number of tokens used for such a simple message. Think about why that might happen and how you could optimise it.

💡 Answer / Suggested Prompt

Why so many tokens? Bob injects its full system prompt (custom instructions, active skills, workspace context) into every message — even a simple hello. The token counter reflects all of that hidden context, not just your two words.

How to optimise: Reduce the number of active skills and custom instructions, keep workspace files small, or switch to a lighter mode (e.g. Ask) that carries a shorter system prompt.

Suggested prompt: How many tokens did my last message use, and what contributed to that count?

Section 2 — Live Data with MCP

  1. 1
    Switch to Ask mode

    Change Bob to Ask mode and type: Who is the Premier League Champion of 2026?

    Notice that Bob cannot answer this correctly — its training data has a knowledge cutoff.

  2. 2
    Think about a solution

    How would you solve the knowledge cutoff problem using MCP tools? Think about what kind of tool would allow Bob to access current data.

💡 Answer / Suggested Prompt

Solution: Add a web-search or browser MCP tool (e.g. the Brave Search MCP or a Fetch/Puppeteer MCP server). This gives Bob the ability to query live sources at runtime, bypassing its training-data cutoff.

Suggested prompt: Search the web for the Premier League Champion of 2026 and tell me who won. (requires a web-search MCP to be active)

Section 3 — Web Scraping

  1. 1
    Scrape a test site

    Ask Bob to scrape and copy the styles from: https://webscraper.io/test-sites

💡 Answer / Suggested Prompt

Suggested prompt: Scrape https://webscraper.io/test-sites, extract the CSS styles used on the page, and reproduce them as a standalone stylesheet I can copy.

Bob will use a Fetch or Puppeteer MCP tool to retrieve the page, then parse and list the stylesheets and inline styles it finds.

Lab 3

Improve Code Quality with Bob

Use Bob's Plan mode and Ponytail slash command to implement a service, catch vulnerabilities, and run a code review on the code you just built.

Same Branch

Labs 2 and 3 share the same branch (lab_2_mcp_and_3_code_quality). You should already be on the correct branch from Lab 2.

Steps

  1. 1
    Switch to Plan mode & implement the API

    Switch Bob to Plan mode, then use the prompt below:

    Plan mode — implement API
    Help implement simple API service for the web you just scraped
  2. 2
    Check Bob Findings for vulnerabilities

    After the implementation is complete, review the Bob Findings panel. If any security vulnerabilities appear, ask Bob to fix them.

  3. 3
    Run Ponytail code review

    Type / in the chatbox to reveal slash commands, then select /ponytail to trigger a Ponytail review of the code you just wrote. If Ponytail does not optimise automatically, ask it to run code optimisation.
